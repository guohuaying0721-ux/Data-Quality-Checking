分组基准机制 · 最小颗粒

不同工位/阶段的"正常"本就不同，统一阈值两头不讨好。所以按 wav 的 .json 边车里的 机台 × 阶段 × 通道 最小颗粒分组，每组用自己的基准比对，学与检完全对称。

分组键 = detectPointName_stage{N}_ch{N}
学检对称
phase2 按此键分组学统计量；phase3 每个待检文件读边车拼出同一键，只取本组基准跑阈值。
严格同组、无兜底：待检文件的组若不在基准里，即使同产品也判 incompatible —— 上新机台/新阶段前须先喂干净样本扩基准。
读 .json 边车→ 拼分组键→ 匹配基准组→ 该组阈值检测
判定流水线 · 早返回，顺序即优先级

phase3 自上而下查，第一条命中的规则即最终标签，每个文件只拿一个标签。分四梯队，梯队之间的顺序是硬约束：先认出确凿的物理硬伤，再问兼不兼容，最后才拿基准量细节。

梯队判定含义命中→
A能否处理文件读不出 / 特征算不了corrupt
B物理硬伤 · 硬规则（全量）空采、削波 —— 与产品无关，抢在兼容门之前，避免被"不兼容"掩盖empty / clipped
C有没有尺子无匹配基准组（跨产品 / 无元数据）→ 后面规则没尺子可比incompatible
D拿尺子量细节 · per-group × CALIB八条基准规则依次查，全过则保留对应异常 / keep

CALIB 校准机制

基准只产"数据长什么样"的原始统计量；"多严格"由一组系数 CALIB 决定。生效阈值 = 基准统计量 × CALIB 系数，所有灵敏度收口一处，调松紧只动这里。

CALIB 的"两段式"行为 · 必须知道
注意
CALIB 有两个身份：① 代码里的定义（真理来源）；② phase2 学基准时整份固化进 thresholds.json 的快照。phase3 检测时优先用 thresholds.json 里的快照。
所以 只改代码里的 CALIB 不会改变检测结果 —— 必须重跑 phase2 重新固化。这是为了可复现：拿单个 thresholds.json 就能精确重现那次检测。
改 CALIB 系数→ 重跑 phase2（重新固化）→ 重跑 phase3 生效
设计原则
共享内核 · 单一定义
IO / 特征提取 / empty / clipped 统一在 common.py，三个 phase 共用。避免规则在各文件复制并发散 —— 同一文件在快筛与复检中结果必然一致。
分级对称 · too_weak ↔ too_loud
能量既有"过弱"也有"过响"，规则成对存在，避免不对称漏检。同产品同组但显著偏离本组水平 → 归类为对应异常，而非误判成"不兼容"。
绝对地板 · 防近零基准塌缩
乘法阈值碰到近零基准（如首尾几乎无静音）会塌缩到 0，任何抖动都误报。对静音类阈值加绝对地板（如 0.5s），低于此一律不算。
